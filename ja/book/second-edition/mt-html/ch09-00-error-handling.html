Error Handlingエラー処理


Rust's commitment to reliability extends to error handling.信頼性へのRustの取り組みは、エラー処理にまで及んでいます。 Errors are a fact of life in software, so Rust has a number of features for handling situations in which something goes wrong.ソフトウェアではエラーは人生の事実なので、Rustには何かがうまくいかない状況を処理するための多くの機能があります。 In many cases, Rust requires you to acknowledge the possibility of an error and take some action before your code will compile.多くの場合、Rustはエラーの可能性を認識し、コードがコンパイルされる前に何らかのアクションをとる必要があります。 This requirement makes your program more robust by ensuring that you'll discover errors and handle them appropriately before you've deployed your code to production!この要件は、コードを本番環境に導入する前に、エラーを発見して適切に処理することによって、プログラムをより堅牢にします。


Rust groups errors into two major categories: recoverable and unrecoverable errors.錆は、エラーを回復可能 なエラーと回復不可能なエラーの2つの主要なカテゴリに分類します。 For a recoverable error, such as a file not found error, it's reasonable to report the problem to the user and retry the operation.ファイルが見つからないなどの回復可能なエラーの場合、問題をユーザーに報告して操作を再試行するのが妥当です。 Unrecoverable errors are always symptoms of bugs, like trying to access a location beyond the end of an array.回復不可能なエラーは、アレイの終わりを超えた場所にアクセスしようとするような、常にバグの症状です。


Most languages don't distinguish between these two kinds of errors and handle both in the same way, using mechanisms such as exceptions.ほとんどの言語は、例外などのメカニズムを使用して、これらの2種類のエラーを区別せず、同じ方法で両方を処理しません。 Rust doesn't have exceptions.錆には例外がありません。 Instead, it has the type Result<T, E> for recoverable errors and the panic€ macro that stops execution when the program encounters an unrecoverable error.その代わりに、回復可能なエラーの場合はResult<T, E> 、回復不可能なエラーが発生した場合は実行を停止するpanic€マクロがあります。 This chapter covers calling panic€ first and then talks about returning Result<T, E> values.この章ではまずpanic€を呼び出す方法について説明し、次にResult<T, E>値を返す方法について説明します。 Additionally, we'll explore considerations when deciding whether to try to recover from an error or to stop execution.さらに、エラーからの回復を試みるか、実行を停止するかを決定する際に考慮すべき点について検討します。
