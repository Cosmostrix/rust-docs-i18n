Appendix G - How Rust is Made and “Nightly Rust”付録G - 錆がどのように作られ、 "夜間の錆"


This appendix is about how Rust is made and how that affects you as a Rust developer.この付録は、Rustがどのように作られたのか、それがRustの開発者としてどのように影響するのかについてです。 We mentioned that the output in this book was generated by stable Rust 1.21.0, but any examples that compile should continue to compile in any stable version of Rust greater than that.この本の出力は安定したRust 1.21.0によって生成されたと述べましたが、それより大きな安定版の安定版であればコンパイルを続けるべきです。 This section is to explain how we ensure this is true!このセクションでは、これが正しいことをどのように保証するかについて説明します。


Stability Without Stagnation停滞のない安定


As a language, Rust cares a lot about the stability of your code.言語として、Rustはコードの安定性について多くのことを心配しています。 We want Rust to be a rock-solid foundation you can build on, and if things were constantly changing, that would be impossible.私たちは、ルストをあなたが上に構築できる堅固な基盤にしたいと思っています。物事が絶えず変化しているのなら、それは不可能でしょう。 At the same time, if we can't experiment with new features, we may not find out important flaws until after their release, when we can no longer change things.同時に、新しい機能を試すことができない場合、リリース後、私たちが物事を変えることができなくなるまで、重要な欠陥を見つけられないことがあります。


Our solution to this problem is what we call “stability without stagnation”, and our guiding principle is this: you should never have to fear upgrading to a new version of stable Rust.この問題に対する当社の解決策は、「停滞のない安定性」と呼ばれるものであり、我々の指針はこれです。安定したRustの新しいバージョンへのアップグレードを恐れる必要はありません。 Each upgrade should be painless, but should also bring you new features, fewer bugs, and faster compile times.各アップグレードは無痛であるべきですが、新機能、バグの削減、コンパイル時間の短縮も必要です。


Choo, Choo!汽車ポッポ！ Release Channels and Riding the Trainsチャネルを解放して列車に乗る


Rust development operates on a train schedule .錆の開発は電車のスケジュールで行われます。 That is, all development is done on the master branch of the Rust repository.つまり、すべての開発はRustリポジトリのmasterブランチで行われます。 Releases follow a software release train model, which has been used by Cisco IOS and other software projects.リリースは、Cisco IOSおよびその他のソフトウェアプロジェクトで使用されているソフトウェアリリーストレインモデルに従います。 There are three release channels for Rust: Rustには3つのリリースチャンネルがあります：


Nightlyナイトリー
Betaベータ
Stable安定

Most Rust developers primarily use the stable channel, but those who want to try out experimental new features may use nightly or beta.ほとんどの錆の開発者は、主に安定したチャネルを使用しますが、実験的な新機能を試してみたい人は、夜間またはベータ版を使用できます。


Here's an example of how the development and release process works: let's assume that the Rust team is working on the release of Rust 1.5. RustチームがRust 1.5のリリースに取り組んでいると仮定してみましょう。 That release happened in December of 2015, but it will provide us with realistic version numbers.このリリースは2015年12月に行われましたが、現実的なバージョン番号を提供します。 A new feature is added to Rust: a new commit lands on the master branch.新しい機能がRustに追加されました。新しいコミットがmasterブランチに着きます。 Each night, a new nightly version of Rust is produced.毎晩新しいRustの夜間版が制作されます。 Every day is a release day, and these releases are created by our release infrastructure automatically.毎日がリリース日です。これらのリリースは、リリースインフラストラクチャによって自動的に作成されます。 So as time passes, our releases look like this, once a night:だから時間がたつにつれて、私たちのリリースは、一晩に一度、このようになります：



Every six weeks, it's time to prepare a new release! 6週間ごとに、新しいリリースを準備する時です！ The beta branch of the Rust repository branches off from the master branch used by nightly. Rustリポジトリのbetaブランチは、夜間に使用されるmasterブランチから分岐します。 Now, there are two releases:今、2つのリリースがあります：



Most Rust users do not use beta releases actively, but test against beta in their CI system to help Rust discover possible regressions.ほとんどのRustユーザーはベータ版を積極的に使用していませんが、Rustが可能な回帰を発見するのを支援するために、CIシステムのベータ版をテストします。 In the meantime, there's still a nightly release every night:それまでの間は、毎晩夜間にリリースされています。



Let's say a regression is found.回帰が見つかったとしましょう。 Good thing we had some time to test the beta release before the regression snuck into a stable release!回帰が安定したリリースになる前に、ベータ版をテストする時間がありました。 The fix is applied to master , so that nightly is fixed, and then the fix is backported to the beta branch, and a new release of beta is produced:この修正はmasterに適用され、毎晩修正され、修正はbetaブランチにバックポートされ、新しいベータ版がリリースされます：



Six weeks after the first beta was created, it's time for a stable release!最初のベータ版が作成されてから6週間後、安定版がリリースされました。 The stable branch is produced from the beta branch: stableブランチはbetaブランチから生成されます：

This is called the “train model” because every six weeks, a release “leaves the station”, but still has to take a journey through the beta channel before it arrives as a stable release.これは「列車モデル」と呼ばれ、6週間ごとにリリースが「駅を離れる」ため、安定版として到着する前にベータ・チャンネルを通る必要があります。


Rust releases every six weeks, like clockwork.錆は時計のように6週間ごとに放出されます。 If you know the date of one Rust release, you can know the date of the next one: it's six weeks later. 1回の錆放出の日付を知っていれば、次の日付を知ることができます.6週間後です。 A nice aspect of having releases scheduled every six weeks is that the next train is coming soon. 6週間ごとにリリースが予定されていることの素晴らしい点は、すぐに次の列車が来ることです。 If a feature happens to miss a particular release, there's no need to worry: another one is happening in a short time!ある機能が特定のリリースを見逃した場合、短期間に別の機能が使用されていることを心配する必要はありません。 This helps reduce pressure to sneak possibly unpolished features in close to the release deadline.これにより、リリースの締め切りに近い時点で未研磨のフィーチャを潜り込ませるというプレッシャーを軽減できます。


Thanks to this process, you can always check out the next build of Rust and verify for yourself that it's easy to upgrade to: if a beta release doesn't work as expected, you can report it to the team and get it fixed before the next stable release happens!このプロセスのおかげで、いつでもRustの次のビルドをチェックして、アップグレードが容易であることを自分自身で確認することができます。ベータ版が期待通りに機能しない場合は、チームに報告し、次の安定リリースが起こります！ Breakage in a beta release is relatively rare, but rustc is still a piece of software, and bugs do exist.ベータリリースでの破損は比較的まれですが、 rustcはまだソフトウェアの1つであり、バグも存在します。


Unstable Features不安定な機能


There's one more catch with this release model: unstable features.このリリースモデルにはもう一つの欠点があります。不安定な機能です。 Rust uses a technique called “feature flags” to determine what features are enabled in a given release. Rustは、特定のリリースでどの機能が有効になっているかを判断するために、「機能フラグ」という技術を使用します。 If a new feature is under active development, it lands on master , and therefore, in nightly, but behind a feature flag .新しい機能が積極的に開発されている場合、それはmasterになるため、夜間に機能フラグの後ろにあります 。 If you, as a user, wish to try out the work-in-progress feature, you can, but you must be using a nightly release of Rust and annotate your source code with the appropriate flag to opt in.ユーザーとして、作業中の機能を試してみることができますが、夜間リリースのRustを使用し、ソースコードに適切なフラグを付けてオプトインする必要があります。


If you're using a beta or stable release of Rust, you can't use any feature flags. Rustのベータ版または安定版を使用している場合、機能フラグは使用できません。 This is the key that allows us to get practical use with new features before we declare them stable forever.これは、永遠に安定して宣言する前に、新しい機能を実用化するための鍵です。 Those who wish to opt into the bleeding edge can do so, and those who want a rock-solid experience can stick with stable and know that their code won't break.出血を望む人はそ​​うすることができ、堅実な経験を望む人は安定していて、コードが壊れないことを知ることができます。 Stability without stagnation.停滞のない安定性。


This book only contains information about stable features, as in-progress features are still changing, and surely they'll be different between when this book was written and when they get enabled in stable builds.この本には安定した機能に関する情報しか含まれていませんが、進行中の機能はまだ変化しています。この本が書かれたときと安定したビルドで有効になる時期は異なっています。 You can find documentation for nightly-only features online.オンラインで夜間のみの機能に関するドキュメントを見つけることができます。

The RFC Process and Teams RFCプロセスとチーム


So how do you learn about these new features?では、これらの新機能についてどのように学びますか？ Rust's development model follows a Request For Comments (RFC) process . Rustの開発モデルはRequest For Comments（RFC）プロセスに従います 。 If you'd like an improvement in Rust, you can write up a proposal, called an RFC. Rustの改善を望むなら、RFCと呼ばれる提案書を書くことができます。


Anyone can write RFCs to improve Rust, and the proposals are reviewed and discussed by the Rust team, which is comprised of many topic subteams.誰でもRustを改善するためのRFCを書くことができ、提案は多くのトピックサブチームで構成されるRustチームによって審査され、議論されます。 There's a full list of the teams on Rust's website , which includes teams for each area of the project: language design, compiler implementation, infrastructure, documentation, and more. Rustのウェブサイトには 、言語デザイン、コンパイラの実装、インフラストラクチャ、ドキュメントなど、プロジェクトの各分野のチームが含まれています。 The appropriate team reads the proposal and the comments, writes some comments of their own, and eventually, there's consensus to accept or reject the feature.適切なチームが提案とコメントを読み、独自のコメントを書き、最終的には機能を受け入れるか拒否するかの合意があります。


If the feature is accepted, an issue is opened on the Rust repository, and someone can implement it.この機能が受け入れられると、Rustリポジトリに問題がオープンされ、誰かがそれを実装することができます。 The person who implements it very well may not be the person who proposed the feature in the first place!それを非常にうまく実装する人は、最初にその機能を提案した人ではないかもしれません！ When the implementation is ready, it lands on the master branch behind a feature gate, as we discussed in the “Unstable Features” section.実装が準備完了すると、「不安定な機能」の項で説明したように、フィーチャーゲートの背後にあるmasterブランチに着陸します。


After some time, once Rust developers who use nightly releases have been able to try out the new feature, team members will discuss the feature, how it's worked out on nightly, and decide if it should make it into stable Rust or not.しばらくすると、夜間リリースを使用しているRustの開発者が新しい機能を試してみると、チームメンバーがその機能について話し合い、夜間にどのように動いているのか、安定したRustにするかどうかを決定します。 If the decision is to move forward, the feature gate is removed, and the feature is now considered stable!意思決定が前進する場合、フィーチャゲートは削除され、フィーチャは安定していると見なされます。 It rides the trains into a new stable release of Rust.それは列車を新しい安定したリリースのRustに乗せます。
