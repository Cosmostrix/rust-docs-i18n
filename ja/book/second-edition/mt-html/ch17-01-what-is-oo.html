Characteristics of Object-Oriented Languagesオブジェクト指向言語の特徴


There is no consensus in the programming community about what features a language must have to be considered object oriented.プログラミングコミュニティには、言語がオブジェクト指向とみなされる必要がある機能についてのコンセンサスはない。 Rust is influenced by many programming paradigms, including OOP;錆は、OOPを含む多くのプログラミングパラダイムの影響を受けます。 for example, we explored the features that came from functional programming in Chapter 13. Arguably, OOP languages share certain common characteristics, namely objects, encapsulation, and inheritance.たとえば、第13章の関数型プログラミングの機能を調べました.OOP言語は、オブジェクト、カプセル化、および継承といった特定の共通の特性を共有しています。 Let's look at what each of those characteristics means and whether Rust supports it.これらの特徴のそれぞれが何を意味し、Rustがそれをサポートしているかを見てみましょう。


Objects Contain Data and Behaviorデータと動作を含むオブジェクト


The book Design Patterns: Elements of Reusable Object-Oriented Software by Enoch Gamma, Richard Helm, Ralph Johnson, and John Vlissides (Addison-Wesley Professional, 1994) colloquially referred to as The Gang of Four book, is a catalog of object-oriented design patterns. 「 Design Patterns： Enoch Gamma、Richard Helm、Ralph Johnson、John Vlissides（Addison-Wesley Professional、1994）の「再利用可能なオブジェクト指向ソフトウェアの要素 」の本は、 The Gang of Fourという本書ではオブジェクト指向のカタログデザインパターン。 It defines OOP this way:これはOOPを以下のように定義します：


Object-oriented programs are made up of objects.オブジェクト指向プログラムはオブジェクトで構成されています。 An object packages both data and the procedures that operate on that data. オブジェクトは 、データとそのデータで動作するプロシージャの両方をパッケージ化します。 The procedures are typically called methods or operations .この手順は、通常、 メソッドまたは操作と呼ばれます 。


Using this definition, Rust is object oriented: structs and enums have data, and impl blocks provide methods on structs and enums.この定義を使用すると、Rustはオブジェクト指向です。構造体と列挙型はデータを持ち、 implブロックは構造体と列挙型のメソッドを提供します。 Even though structs and enums with methods aren't called objects, they provide the same functionality, according to the Gang of Four's definition of objects.メソッドを持つ構造体とenumはオブジェクトと呼ばれませんが、Gang of Fourのオブジェクト定義によれば、同じ機能を提供します。


Encapsulation that Hides Implementation Details実装の詳細を隠すカプセル化


Another aspect commonly associated with OOP is the idea of encapsulation , which means that the implementation details of an object aren't accessible to code using that object.一般的にOOPに関連するもう1つの側面は、オブジェクトの実装の詳細がそのオブジェクトを使用するコードにアクセスできないことを意味するカプセル化のアイデアです。 Therefore, the only way to interact with an object is through its public API;したがって、オブジェクトとやり取りする唯一の方法は、公開APIを使用することです。 code using the object shouldn't be able to reach into the object's internals and change data or behavior directly.オブジェクトを使用するコードは、オブジェクトの内部に到達してデータやビヘイビアを直接変更することはできません。 This enables the programmer to change and refactor an object's internals without needing to change the code that uses the object.これにより、プログラマは、オブジェクトを使用するコードを変更することなく、オブジェクトの内部を変更してリファクタリングすることができます。


We discussed how to control encapsulation in Chapter 7: we can use the pub keyword to decide which modules, types, functions, and methods in our code should be public, and by default everything else is private.第7章でカプセル化を制御する方法について説明しましたpubキーワードを使用して、コード内のどのモジュール、タイプ、関数、メソッドを公開するかを決めることができます。 For example, we can define a struct AveragedCollection that has a field containing a vector of i32 values.たとえば、 i32値のベクトルを含むフィールドを持つstruct AveragedCollectionを定義できます。 The struct can also have a field that contains the average of the values in the vector, meaning the average doesn't have to be computed on demand whenever anyone needs it.構造体には、ベクトルの値の平均を含むフィールドもあります。つまり、必要に応じて、必要に応じて平均値を計算する必要はありません。 In other words, AveragedCollection will cache the calculated average for us.つまり、 AveragedCollectionは計算された平均をキャッシュします。 Listing 17-1 has the definition of the AveragedCollection struct:リスト17-1は、 AveragedCollection構造体の定義を示しています。

If encapsulation is a required aspect for a language to be considered object oriented, then Rust meets that requirement.カプセル化が、言語がオブジェクト指向とみなされるために必要な側面である場合、Rustはその要件を満たします。 The option to use pub or not for different parts of code enables encapsulation of implementation details.コードのさまざまな部分にpubを使用するかどうかを選択すると、実装の詳細をカプセル化できます。


Inheritance as a Type System and as Code Sharingタイプシステムとしての継承とコード共有


Inheritance is a mechanism whereby an object can inherit from another object's definition, thus gaining the parent object's data and behavior without you having to define them again. 継承は、オブジェクトが別のオブジェクトの定義から継承することができるメカニズムであり、そのため、再度定義することなく、親オブジェクトのデータと動作が得られます。


If a language must have inheritance to be an object-oriented language, then Rust is not one.言語がオブジェクト指向言語であるために継承を持たなければならない場合、Rustは1ではありません。 There is no way to define a struct that inherits the parent struct's fields and method implementations.親構造体のフィールドとメソッドの実装を継承する構造体を定義する方法はありません。 However, if you're used to having inheritance in your programming toolbox, you can use other solutions in Rust, depending on your reason for reaching for inheritance in the first place.しかし、プログラミングツールボックスに継承されることに慣れている場合は、最初に継承の理由に応じて、Rustの他のソリューションを使用することができます。


You choose inheritance for two main reasons.あなたは、主に2つの理由から継承を選択します。 One is for reuse of code: you can implement particular behavior for one type, and inheritance enables you to reuse that implementation for a different type. 1つはコードの再利用です。あるタイプに対して特定の動作を実装することができ、継承を使用すると、そのタイプの実装を再利用できます。 You can share Rust code using default trait method implementations instead, which you saw in Listing 10-14 when we added a default implementation of the summarize method on the Summary trait.代わりに、デフォルトの特性メソッド実装を使用してRustコードを共有することができます。これは、リスト10-14でSummary特性のsummarizeメソッドのデフォルト実装を追加したときに見ました。 Any type implementing the Summary trait would have the summarize method available on it without any further code.実装する任意のタイプのSummary特性は持っているでしょうsummarize任意の更なるコードなしにそれに方法が利用できます。 This is similar to a parent class having an implementation of a method and an inheriting child class also having the implementation of the method.これは、メソッドの実装を持つ親クラスと、メソッドの実装を持つ継承する子クラスに似ています。 We can also override the default implementation of the summarize method when we implement the Summary trait, which is similar to a child class overriding the implementation of a method inherited from a parent class.また、 Summaryクラスの実装時には、親クラスから継承したメソッドの実装をオーバーライドする子クラスに似ているSummaryプロパティを実装するときに、 summarizeメソッドのデフォルトの実装をオーバーライドすることもできます。


The other reason to use inheritance relates to the type system: to enable a child type to be used in the same places as the parent type.継承を使用するもう一つの理由は、型のシステムに関係します。子型を親型と同じ場所で使用できるようにすること。 This is also called polymorphism , which means that you can substitute multiple objects for each other at runtime if they share certain characteristics.これは、 ポリモフィズムとも呼ばれ、特定の特性を共有する複数のオブジェクトを実行時に互いに置き換えることができます。


Polymorphism多型


To many people, polymorphism is synonymous with inheritance.多くの人にとって、多型は継承と同義です。 But it's actually a more general concept that refers to code that can work with data of multiple types.しかし、実際には、複数のタイプのデータで動作するコードを指す、より一般的な概念です。 For inheritance, those types are generally subclasses.継承の場合、これらの型は一般にサブクラスです。


Rust instead uses generics to abstract over different possible types and trait bounds to impose constraints on what those types must provide.代わりにRustはジェネリックスを使用して、異なるタイプの可能性のあるタイプと特性境界を抽象化して、それらのタイプが提供しなければならないものに制約を課します。 This is sometimes called bounded parametric polymorphism .これは、 有界パラメトリック多型とも呼ばれます 。


Inheritance has recently fallen out of favor as a programming design solution in many programming languages because it's often at risk of sharing more code than necessary.継承は、多くの場合、必要以上に多くのコードを共有するリスクがあるため、多くのプログラミング言語のプログラミング設計ソリューションとして、近年好まれていません。 Subclasses shouldn't always share all characteristics of their parent class but will do so with inheritance.サブクラスは、親クラスのすべての特性を常に共有してはいけませんが、継承ではそうします。 This can make a program's design less flexible.これにより、プログラムのデザインの柔軟性が低下する可能性があります。 It also introduces the possibility of calling methods on subclasses that don't make sense or that cause errors because the methods don't apply to the subclass.また、メソッドがサブクラスに適用されないため、意味を持たないサブクラスやエラーの原因となるサブクラスでメソッドを呼び出す可能性があります。 In addition, some languages will only allow a subclass to inherit from one class, further restricting the flexibility of a program's design.さらに、言語によっては、サブクラスが1つのクラスから継承することしかできないため、プログラムの設計の柔軟性がさらに制限されます。


For these reasons, Rust takes a different approach, using trait objects instead of inheritance.これらの理由から、Rustは、継承の代わりに形質オブジェクトを使用する別のアプローチを採用しています。 Let's look at how trait objects enable polymorphism in Rust. Rustでどのように形質オブジェクトが多型を可能にするかを見てみましょう。
