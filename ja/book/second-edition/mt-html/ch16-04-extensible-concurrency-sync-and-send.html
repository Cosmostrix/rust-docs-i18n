Extensible Concurrency with the Sync and Send Traits SyncとSend特性による拡張可能な並行性


Interestingly, the Rust language has very few concurrency features.興味深いことに、Rust言語には並行処理機能はほとんどありません。 Almost every concurrency feature we've talked about so far in this chapter has been part of the standard library, not the language.これまでのところ、この章で説明したほぼすべての並行処理機能は、言語ではなく標準ライブラリの一部です。 Your options for handling concurrency are not limited to the language or the standard library;並行処理のオプションは、言語や標準ライブラリに限定されません。 you can write your own concurrency features or use those written by others.あなた自身の並行処理機能を書くことも、他の人が書いたものを使うこともできます。


However, two concurrency concepts are embedded in the language: the std::marker traits Sync and Send .しかし、言語には2つの並行性の概念が組み込まれています： std::marker SyncとSend特徴をSendます。


Allowing Transference of Ownership Between Threads with Send Sendスレッド間の所有権の転送を許可する


The Send marker trait indicates that ownership of the type implementing Send can be transferred between threads. Sendマーカー特性は、 Sendを実装する型の所有権をスレッド間で転送できることを示します。 Almost every Rust type is Send , but there are some exceptions, including Rc<T> : this cannot be Send because if you cloned an Rc<T> value and tried to transfer ownership of the clone to another thread, both threads might update the reference count at the same time.ほぼすべての錆の種類があるSendが、を含むいくつかの例外があるRc<T>これはできませんSendますが、クローン化された場合ので、 Rc<T>値を、別のスレッドへのクローンの所有権を譲渡しようとした、両方のスレッドが更新かもしれません同時に参照カウント。 For this reason, Rc<T> is implemented for use in single-threaded situations where you don't want to pay the thread-safe performance penalty.このため、 Rc<T>は、スレッドセーフなパフォーマンス上のペナルティを支払う必要がないシングルスレッドの状況で使用するために実装されています。


Therefore, Rust's type system and trait bounds ensure that you can never accidentally send an Rc<T> value across threads unsafely.したがって、Rustのタイプシステムと特性境界は、誤ってスレッド間でRc<T>値を誤って送信することがないようにします。 When we tried to do this in Listing 16-14, we got the error the trait Send is not implemented for Rc<Mutex<i32>> .リスト16-14でこれを実行しようとしたときthe trait Send is not implemented for Rc<Mutex<i32>>というエラーが発生しましthe trait Send is not implemented for Rc<Mutex<i32>> 。 When we switched to Arc<T> , which is Send , the code compiled. SendであるArc<T>切り替えると、コードがコンパイルされました。


Any type composed entirely of Send types is automatically marked as Send as well. Sendタイプから完全に構成されたタイプは自動的にSendとマークされます。 Almost all primitive types are Send , aside from raw pointers, which we'll discuss in Chapter 19.ほとんどすべてのプリミティブ型はRawポインタ以外のSendです。これについては第19章で説明します。


Allowing Access from Multiple Threads with Sync Sync複数のスレッドからのアクセスを許可する


The Sync marker trait indicates that it is safe for the type implementing Sync to be referenced from multiple threads. Syncマーカーの特性は、 Syncを実装するタイプが複数のスレッドから参照されることが安全であることを示します。 In other words, any type T is Sync if &T (a reference to T ) is Send , meaning the reference can be sent safely to another thread.言い換えれば、 &T （ Tへの参照）がSendであれば、どのタイプのTもSyncです。これは、参照が別のスレッドに安全に送信できることを意味します。 Similar to Send , primitive types are Sync , and types composed entirely of types that are Sync are also Sync .同様にSend 、原始的な種類がありSync 、および完全にされている種類で構成されるタイプのSyncまたあるSync 。


The smart pointer Rc<T> is also not Sync for the same reasons that it's not Send .スマートポインタRc<T>もSendはないのと同じ理由でSyncではありません。 The RefCell<T> type (which we talked about in Chapter 15) and the family of related Cell<T> types are not Sync . RefCell<T>型（第15章で説明した）と関連するCell<T>型のファミリはSync型ではありません。 The implementation of borrow checking that RefCell<T> does at runtime is not thread-safe. RefCell<T>が実行時に行う借用チェックの実装は、スレッドセーフではありません。 The smart pointer Mutex<T> is Sync and can be used to share access with multiple threads as you saw in the “Sharing a Mutex<T> Between Multiple Threads” section.スマートポインタMutex<T>はSyncあり、「複数のスレッド間のMutex<T>共有」のセクションで見たように、複数のスレッドとのアクセスを共有するために使用できます。


Implementing Send and Sync Manually Is Unsafe SendとSync手動で実装するのが安全でない


Because types that are made up of Send and Sync traits are automatically also Send and Sync , we don't have to implement those traits manually. SendとSync特性で構成されているタイプは自動的にSendとSyncでもあるため、これらの特性を手動で実装する必要はありません。 As marker traits, they don't even have any methods to implement.マーカーの形質としては、実装する方法さえありません。 They're just useful for enforcing invariants related to concurrency.これらは、並行性に関連する不変条件を強制するのに役立ちます。


Manually implementing these traits involves implementing unsafe Rust code.これらの特性を手動で実装するには、安全でない錆のコードを実装する必要があります。 We'll talk about using unsafe Rust code in Chapter 19;第19章では安全でない錆のコードを使用する方法について説明します。 for now, the important information is that building new concurrent types not made up of Send and Sync parts requires careful thought to uphold the safety guarantees.今のところ重要な情報は、 SendとSync部分で構成されていない新しい並行タイプを構築するには、安全の保証を守るために慎重な考えが必要だということです。 The Rustonomicon has more information about these guarantees and how to uphold them. The Rustonomiconは、これらの保証についてのより多くの情報と、それらを維持する方法があります。

As mentioned earlier, because very little of how Rust handles concurrency is part of the language, many concurrency solutions are implemented as crates.前述したように、Rustが並行処理をどのように処理するかは言語の一部であるため、多くの並行処理ソリューションがクレートとして実装されています。 These evolve more quickly than the standard library, so be sure to search online for the current, state-of-the-art crates to use in multithreaded situations.これらは標準ライブラリよりも迅速に進化していますので、マルチスレッドの状況で使用する最新のクレートをオンラインで検索してください。


The Rust standard library provides channels for message passing and smart pointer types, such as Mutex<T> and Arc<T> , that are safe to use in concurrent contexts. Rust標準ライブラリは、並行コンテキストで安全に使用できるMutex<T>やArc<T>などのメッセージパッシングとスマートポインタ型のチャネルを提供します。 The type system and the borrow checker ensure that the code using these solutions won't end up with data races or invalid references.タイプシステムと貸借チェッカーは、これらのソリューションを使用するコードがデータ競合や無効な参照で終わらないことを保証します。 Once you get your code to compile, you can rest assured that it will happily run on multiple threads without the kinds of hard-to-track-down bugs common in other languages.コードをコンパイルすると、他の言語でよく見られるような追跡が困難なバグがなくても、複数のスレッドでうまく動作することが保証されます。 Concurrent programming is no longer a concept to be afraid of: go forth and make your programs concurrent, fearlessly!並行プログラミングは、もはや恐れているコンセプトではありません。あなたのプログラムを、恐れのないように並行させてください！
