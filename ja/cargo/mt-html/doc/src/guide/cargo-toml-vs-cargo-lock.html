Cargo.toml vs Cargo.lock Cargo.tomlとCargo.lock


Cargo.toml and Cargo.lock serve two different purposes. Cargo.tomlとCargo.lockは2つの異なる目的を果たします。 Before we talk about them, here's a summary:私たちがそれらについて話す前に、ここに要約があります：


Cargo.toml is about describing your dependencies in a broad sense, and is written by you. Cargo.tomlは広義の依存関係を記述することについて書かれており、あなたによって書かれています。
Cargo.lock contains exact information about your dependencies. Cargo.lockは、依存関係に関する正確な情報が含まれています。 It is maintained by Cargo and should not be manually edited.これはCargoによって管理されており、手動で編集しないでください。

If you're building a library that other projects will depend on, put Cargo.lock in your .gitignore .他のプロジェクトが依存するライブラリを構築する場合は、 .gitignoreにCargo.lockします。 If you're building an executable like a command-line tool or an application, check Cargo.lock into git .コマンドラインツールやアプリケーションのような実行可能ファイルをビルドする場合は、 Cargo.lockをgitにチェックインしてください。 If you're curious about why that is, see "Why do binaries have Cargo.lock in version control, but not libraries?" 理由が分からないのであれば、 「バイナリにはバージョンコントロールでCargo.lockがあり、ライブラリではないのはなぜですか？」を参照してください。 in the FAQ . よくある質問です。


Let's dig in a little bit more.もう少し詳しく調べてみましょう。


Cargo.toml is a manifest file in which we can specify a bunch of different metadata about our project. Cargo.tomlは、プロジェクトに関するさまざまなメタデータを指定できるマニフェストファイルです。 For example, we can say that we depend on another project:たとえば、別のプロジェクトに依存していると言えます。



This project has a single dependency, on the rand library.このプロジェクトは、 randライブラリに依存しています。 We've stated in this case that we're relying on a particular Git repository that lives on GitHub.このケースでは、GitHubに存在する特定のGitリポジトリに依存していると述べました。 Since we haven't specified any other information, Cargo assumes that we intend to use the latest commit on the master branch to build our project.他の情報を指定していないので、Cargoはプロジェクトを構築するためにmasterブランチで最新のコミットを使用する予定であると想定しています。


Sound good?いいね？ Well, there's one problem: If you build this project today, and then you send a copy to me, and I build this project tomorrow, something bad could happen.さて、1つの問題があります。今日このプロジェクトを構築してから、私にコピーを送って、明日このプロジェクトをビルドすると、何か悪いことが起こる可能性があります。 There could be more commits to rand in the meantime, and my build would include new commits while yours would not.その間にrandへのコミットが増える可能性があり、あなたのビルドには新しいコミットが含まれますが、あなたのものはそうではありません。 Therefore, we would get different builds.したがって、我々は異なるビルドを得るだろう。 This would be bad because we want reproducible builds.再現可能なビルドが必要なので、これは悪いことです。
