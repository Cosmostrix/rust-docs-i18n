The version field versionフィールド


Cargo bakes in the concept of Semantic Versioning , so make sure you follow some basic rules:カーゴはSemantic Versioningの概念を喚起しているので、基本的なルールに従ってください。


Before you reach 1.0.0, anything goes, but if you make breaking changes, increment the minor version. 1.0.0に達する前に、何かが行きますが、壊れた変更を加えた場合はマイナーバージョンを増やしてください。 In Rust, breaking changes include adding fields to structs or variants to enums. Rustでは、構造体やバリアントへのフィールドを列挙型に追加するなどの変更が行われました。
After 1.0.0, only make breaking changes when you increment the major version. 1.0.0以降、メジャーバージョンをインクリメントするときにのみ改変します。 Don't break the build.ビルドを壊さないでください。
After 1.0.0, don't add any new public API (no new pub anything) in tiny versions. 1.0.0以降、小さなパブリックAPIを追加しないでください（新しいpubません）。 Always increment the minor version if you add any new pub structs, traits, fields, types, functions, methods or anything else.新しいpub構造体、特性、フィールド、型、関数、メソッドなどを追加する場合は、常にマイナーバージョンをインクリメントします。
Use version numbers with three numeric parts such as 1.0.0 rather than 1.0. 1.0ではなく1.0.0などの3つの数値部分を持つバージョン番号を使用します。

The exclude and include fields (optional) excludeフィールドとincludeフィールド（オプション）


You can explicitly specify to Cargo that a set of globs should be ignored or included for the purposes of packaging and rebuilding a package.パッケージをパッケージ化して再構築する目的で、一連のglobsを無視または含めるべきであることをCargoに明示的に指定することができます。 The globs specified in the exclude field identify a set of files that are not included when a package is published as well as ignored for the purposes of detecting when to rebuild a package, and the globs in include specify files that are explicitly included.で指定されたグロブexcludeフィールドは、パッケージが検出の目的のために公開だけでなく、無視されたときに、パッケージを再構築する際には含まれていないファイルのセットを識別し、そして中グロブをinclude 、明示的に含まれるファイルを指定します。


If a VCS is being used for a package, the exclude field will be seeded with the VCS' ignore settings ( .gitignore for git for example). VCSがパッケージに使用されている場合、 excludeフィールドにはVCSの無視設定（gitの場合は.gitignoreなど）が設定されます。




The options are mutually exclusive: setting include will override an exclude .オプションは相互に排他的です： includeを設定するとexcludeが上書きされます。 Note that include must be an exhaustive list of files as otherwise necessary source files may not be included. includeはファイルの完全なリストでなければならないことに注意してください。そうしないと、必要なソースファイルが含まれない可能性があります。

Rulesルール


The usage of features is subject to a few rules:機能の使用にはいくつかのルールがあります：


Feature names must not conflict with other package names in the manifest.機能名は、マニフェスト内の他のパッケージ名と競合してはなりません。 This is because they are opted into via features = [...] , which only has a single namespace.これは、 features = [...]経由でオプトインされているためfeatures = [...]これには、単一の名前空間しかありません。
With the exception of the default feature, all features are opt-in. default機能を除いて、すべての機能はオプトインです。 To opt out of the default feature, use default-features = false and cherry-pick individual features.デフォルト機能をオプトアウトするには、default default-features = falseを使用し、cherry-pick個々の機能を使用します。
Feature groups are not allowed to cyclically depend on one another.フィーチャグループは、周期的に相互に依存することはできません。
Dev-dependencies cannot be optional. dev依存関係はオプションではありません。
Features groups can only reference optional dependencies.機能グループはオプションの依存関係のみを参照できます。
When a feature is selected, Cargo will call rustc with --cfg feature="${feature_name}" .機能を選択した場合は、カーゴは、呼び出しますrustcで--cfg feature="${feature_name}" 。 If a feature group is included, it and all of its individual features will be included.機能グループが含まれている場合、その機能グループとその個々の機能すべてが含まれます。 This can be tested in code via #[cfg(feature = "foo")] .これは#[cfg(feature = "foo")]介してコードでテストできます。

Note that it is explicitly allowed for features to not actually activate any optional dependencies.フィーチャが実際にオプションの依存関係をアクティブ化しないことは、明示的に許可されていることに注意してください。 This allows packages to internally enable/disable features without requiring a new dependency.これにより、パッケージは新しい依存関係を必要とせずに機能を内部で有効/無効にすることができます。


Usage in end products最終製品での使用


One major use-case for this feature is specifying optional features in end-products.この機能の主なユースケースの1つは、最終製品のオプション機能の指定です。 For example, the Servo project may want to include optional features that people can enable or disable when they build it.たとえば、Servoプロジェクトでは、ユーザーがビルド時に有効または無効にできるオプションの機能を含めることができます。


In that case, Servo will describe features in its Cargo.toml and they can be enabled using command-line flags:その場合、ServoはCargo.toml機能を記述し、コマンドラインフラグを使用して有効にすることができます。



Default features could be excluded using --no-default-features .デフォルトの機能は、 --no-default-featuresを使用して除外でき--no-default-features 。


Usage in packagesパッケージでの使用


In most cases, the concept of optional dependency in a library is best expressed as a separate package that the top-level application depends on.ほとんどの場合、ライブラリ内のオプションの依存関係の概念は、トップレベルのアプリケーションが依存する別個のパッケージとして最もよく表現されます。


However, high-level packages, like Iron or Piston, may want the ability to curate a number of packages for easy installation.しかし、IronやPistonのような高水準のパッケージでは、簡単にインストールできるように多数のパッケージを管理する必要があります。 The current Cargo system allows them to curate a number of mandatory dependencies into a single package for easy installation.現在のCargoシステムでは、多数の必須の依存関係を1つのパッケージにまとめることができ、簡単なインストールが可能です。


In some cases, packages may want to provide additional curation for optional dependencies:いくつかのケースでは、パッケージは、オプションの依存関係の追加のキュレーションを提供したいかもしれません：


grouping a number of low-level optional dependencies together into a single high-level feature;複数の低レベルのオプションの依存関係を1つのハイレベル機能にグループ化します。
specifying packages that are recommended (or suggested) to be included by users of the package;パッケージのユーザーがパッケージに含めることを推奨（または提案）するパッケージを指定する。 andそして
including a feature (like secure-password in the motivating example) that will only work if an optional dependency is available, and would be difficult to implement as a separate package (for example, it may be overly difficult to design an IO package to be completely decoupled from OpenSSL, with opt-in via the inclusion of a separate package).オプションの依存関係が利用可能である場合にのみ機能し、別個のパッケージとして実装するのが難しい機能（例えば、動機付けの例ではsecure-passwordような）を含む（IOパッケージを設計するのは非常に困難かもしれないOpenSSLから完全に切り離され、別のパッケージを含めることでオプトインされます）。

In almost all cases, it is an antipattern to use these features outside of high-level packages that are designed for curation.ほとんどすべてのケースで、キュレーションのために設計された高水準パッケージの外でこれらの機能を使用するのは反パターンです。 If a feature is optional, it can almost certainly be expressed as a separate package.機能がオプションである場合は、ほぼ確実に別のパッケージとして表現できます。

The root crate 's Cargo.toml contains the [workspace] table, but is not required to have other configuration. ルートクレートのCargo.tomlには[workspace]テーブルが含まれてい[workspace]が、他の設定は必要ありません。

RFC 1525

The root crate of a workspace, indicated by the presence of [workspace] in its manifest, is responsible for defining the entire workspace.マニフェストに[workspace]が存在することで示されるワークスペースのルートボックスは、ワークスペース全体の定義を担当します。 All path dependencies residing in the workspace directory become members.ワークスペースディレクトリに存在するすべてのpath依存関係がメンバーになります。 You can add additional packages to the workspace by listing them in the members key.パッケージをワークスペースに追加するには、それらをmembersキーにリストします。 Note that members of the workspaces listed explicitly will also have their path dependencies included in the workspace.明示的にリストされたワークスペースのメンバーも、ワークスペースにパスの依存関係が含まれていることに注意してください。 Sometimes a project may have a lot of workspace members and it can be onerous to keep up to date.プロジェクトには多くのワークスペースメンバーが存在することがあり、最新の状態に保つのは面倒です。 The path dependency can also use globs to match multiple paths.パスの依存関係は、複数のパスにマッチするためにglobsを使用することもできます。 Finally, the exclude key can be used to blacklist paths from being included in a workspace.最後に、 excludeキーを使用してパスがワークスペースに含まれるのをブラックリストに登録することができます。 This can be useful if some path dependencies aren't desired to be in the workspace at all.これは、パスの依存関係が作業領域内にあることが望ましくない場合に便利です。


The package.workspace manifest key (described above) is used in member crates to point at a workspace's root crate. package.workspaceマニフェストキー（前述）は、ワークスペースのルートクレートをポイントするためにメンバークレートで使用されます。 If this key is omitted then it is inferred to be the first crate whose manifest contains [workspace] upwards in the filesystem.このキーが省略されている場合、マニフェストはファイルシステム内で[workspace]上に含む最初の木枠であると推定されます。


A crate may either specify package.workspace or specify [workspace] .クレートはpackage.workspace指定するか、 [workspace]指定します。 That is, a crate cannot both be a root crate in a workspace (contain [workspace] ) and also be a member crate of another workspace (contain package.workspace ).つまり、クレートは、ワークスペース（ [workspace]含む）と他のワークスペースのメンバークレート（ package.workspace含む）の両方に存在することはできません。


Most of the time workspaces will not need to be dealt with as cargo new and cargo init will handle workspace configuration automatically.ほとんどの場合、 cargo newおよびcargo initが自動的に作業領域の設定を処理するため、作業領域を処理する必要はありません。

The project layoutプロジェクトのレイアウト


If your project is an executable, name the main source file src/main.rs .プロジェクトが実行可能ファイルの場合は、メインソースファイルの名前をsrc/main.rsます。 If it is a library, name the main source file src/lib.rs .ライブラリの場合は、メインソースファイルsrc/lib.rsに名前を付けます。
