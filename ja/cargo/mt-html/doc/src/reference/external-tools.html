External tools外部ツール


One of the goals of Cargo is simple integration with third-party tools, like IDEs and other build systems. Cargoの目標の1つは、IDEやその他のビルドシステムなどのサードパーティツールとのシンプルな統合です。 To make integration easier, Cargo has several facilities:統合を容易にするために、Cargoにはいくつかの機能があります。


a cargo metadata command, which outputs project structure and dependencies information in JSON,プロジェクト構造と依存関係情報をJSONで出力するcargo metadataコマンド、

a --message-format flag, which outputs information about a particular build, and --message-formatフラグは、特定のビルドに関する情報を出力します。

support for custom subcommands.カスタムサブコマンドのサポート


Information about project structureプロジェクトの構造に関する情報


You can use cargo metadata command to get information about project structure and dependencies. cargo metadataコマンドを使用して、プロジェクトの構造と依存性に関する情報を取得できます。 The output of the command looks like this:コマンドの出力は次のようになります。


Integer version number of the format.フォーマットの整数バージョン番号。
List of packages for this workspace, including dependencies.依存関係を含むこのワークスペースのパッケージのリスト。
Opaque package identifier.不透明なパッケージ識別子。
A list of declared dependencies, see resolve field for actual dependencies.宣言された依存関係のリスト。実際の依存関係のresolveフィールドを参照してください。
Path to Cargo.toml Cargo.tomlへの道
Dependencies graph.依存関係グラフ。

The format is stable and versioned.フォーマットは安定しており、バージョン管理されています。 When calling cargo metadata , you should pass --format-version flag explicitly to avoid forward incompatibility hazard. cargo metadata呼び出すときは、前方互換性の危険を避けるために--format-versionフラグを明示的に渡す必要があります。


If you are using Rust, there is cargo_metadata crate. Rustを使用している場合、 cargo_metadataがあります。


cargo_metadata

Information about buildビルドに関する情報


When passing --message-format=json , Cargo will output the following information during the build: --message-format=jsonを渡すと、Cargoはビルド中に以下の情報を出力します：


compiler errors and warnings,コンパイラのエラーと警告、

produced artifacts,生成されたアーチファクト、

results of the build scripts (for example, native dependencies).ビルドスクリプトの結果（たとえば、ネイティブ依存関係）。

The output goes to stdout in the JSON object per line format.出力はJSONオブジェクトの行形式ごとにstdoutに出力されます。 The reason field distinguishes different kinds of messages. reasonフィールドは、さまざまな種類のメッセージを区別します。


Information about dependencies in the Makefile-compatible format is stored in the .d files alongside the artifacts. Makefileと互換性のある形式の依存関係に関する情報は、アーティファクトとともに.dファイルに格納されます。



Custom subcommandsカスタムサブコマンド


Cargo is designed to be extensible with new subcommands without having to modify Cargo itself. Cargoは、貨物自体を変更することなく、新しいサブコマンドで拡張できるように設計されています。 This is achieved by translating a cargo invocation of the form cargo (?<command>[^ ]+) into an invocation of an external tool cargo-${command} .これは、貨物(?<command>[^ ]+)という形式の貨物呼び出しを外部ツールcargo-${command}呼び出しに変換することによって実現されます。 The external tool must be present in one of the user's $PATH directories.外部ツールは、ユーザーの$PATHディレクトリの1つに存在する必要があります。


When Cargo invokes a custom subcommand, the first argument to the subcommand will be the filename of the custom subcommand, as usual. Cargoがカスタムサブコマンドを呼び出すと、サブコマンドの最初の引数は、通常どおりカスタムサブコマンドのファイル名になります。 The second argument will be the subcommand name itself. 2番目の引数は、サブコマンド名そのものです。 For example, the second argument would be ${command} when invoking cargo-${command} .たとえば、2番目の引数は${command} cargo-${command}を呼び出すときです。 Any additional arguments on the command line will be forwarded unchanged.コマンドラインの追加の引数は変更されずに転送されます。


Cargo can also display the help output of a custom subcommand with cargo help ${command} .貨物は、 cargo help ${command}を使用してカスタムサブコマンドのヘルプ出力を表示することもできます。 Cargo assumes that the subcommand will print a help message if its third argument is --help . Cargoは、3番目の引数が--help場合、サブコマンドはヘルプメッセージを出力することを前提としてい--help 。 So, cargo help ${command} would invoke cargo-${command} ${command} --help .ですから、 cargo help ${command}はcargo-${command} ${command} --helpを呼び出しcargo-${command} ${command} --help 。


Custom subcommands may use the CARGO environment variable to call back to Cargo.カスタムサブコマンドは、 CARGO環境変数を使用してCargoにコールバックすることができます。 Alternatively, it can link to cargo crate as a library, but this approach has drawbacks:あるいは、ライブラリとしてcargo箱にリンクすることもできますが、この方法には欠点があります。


Cargo as a library is unstable: the API may change without deprecationライブラリとしての貨物は不安定です：APIは廃止されることなく変更されることがあります

versions of the linked Cargo library may be different from the Cargo binaryリンクされた貨物ライブラリのバージョンは、貨物バイナリと異なる場合があります
