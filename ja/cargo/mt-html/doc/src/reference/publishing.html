Publishing on crates.io crates.ioでの公開


Once you've got a library that you'd like to share with the world, it's time to publish it on crates.io !あなたが世界と共有したいライブラリを手に入れたら、それをcrates.ioに公開するcrates.ioです！ Publishing a crate is when a specific version is uploaded to be hosted on crates.io .クレートを公開するのは、 crates.ioでホストされる特定のバージョンがアップロードされたcrates.ioです。


Take care when publishing a crate, because a publish is permanent .パブリッシュは永久的なので、木枠のパブリッシュ時には注意してください。 The version can never be overwritten, and the code cannot be deleted.バージョンを上書きすることはできず、コードを削除することはできません。 There is no limit to the number of versions which can be published, however.ただし、発行可能なバージョンの数に制限はありません。


Before your first publish最初の公開前


First thing's first, you'll need an account on crates.io to acquire an API token.まず最初に、APIトークンを取得するにはcrates.ioアカウントが必要です。 To do so, visit the home page and log in via a GitHub account (required for now).これを行うには、 ホームページにアクセスし、GitHubアカウント経由でログインしてください （今のところ必要です）。 After this, visit your Account Settings page and run the cargo login command specified.その後、 Account Settingsページにアクセスし、指定したcargo loginコマンドを実行します。



This command will inform Cargo of your API token and store it locally in your ~/.cargo/credentials (previously it was ~/.cargo/config ).このコマンドはCargoにあなたのAPIトークンを通知し、 ~/.cargo/credentials （以前は~/.cargo/config ）にローカルに格納します。 Note that this token is a secret and should not be shared with anyone else.このトークンは秘密であり、他の誰とも共有してはならないことに注意してください。 If it leaks for any reason, you should regenerate it immediately.それが何らかの理由で漏れた場合は、直ちに再生する必要があります。


Before publishing a new crate新しいクレートを公開する前に


Keep in mind that crate names on crates.io are allocated on a first-come-first- serve basis. crates.io木枠名は、 crates.ioに割り当てられます。 Once a crate name is taken, it cannot be used for another crate.いったん木箱の名前が取られると、別の木箱には使用できません。


Packaging a crate箱の梱包


The next step is to package up your crate into a format that can be uploaded to crates.io .次のステップは、あなたのクレートをcrates.ioにアップロードできる形式にパッケージ化することです。 For this we'll use the cargo package subcommand.このために、 cargo packageサブコマンドを使用しcargo package 。 This will take our entire crate and package it all up into a *.crate file in the target/package directory.これは私たちの全体の箱を取り、それをすべてtarget/packageディレクトリの*.crateファイルにtarget/package 。



As an added bonus, the *.crate will be verified independently of the current source tree.追加ボーナスとして、 *.crateは現在のソースツリーとは独立して検証されます。 After the *.crate is created, it's unpacked into target/package and then built from scratch to ensure that all necessary files are there for the build to succeed. *.crateが作成されると、それはtarget/packageに展開され、最初から構築され、ビルドが成功するために必要なすべてのファイルが存在することを確認します。 This behavior can be disabled with the --no-verify flag.この動作は--no-verifyフラグで無効にすることができます。


Now's a good time to take a look at the *.crate file to make sure you didn't accidentally package up that 2GB video asset, or large data files used for code generation, integration tests, or benchmarking.今度は、 *.crateファイルを見て、誤って2GBのビデオアセットや、コード生成、統合テスト、またはベンチマークに使用される大きなデータファイルをパッケージ化していないことを確認してください。 There is currently a 10MB upload size limit on *.crate files. *.crateファイルには現在10MBのアップロードサイズ制限があります。 So, if the size of tests and benches directories and their dependencies are up to a couple of MBs, you can keep them in your package;したがって、 testsとbenchesディレクトリとその依存関係のサイズが最大2 MBの場合、それらをパッケージに入れておくことができます。 otherwise, better to exclude them.さもなければ、それらを除外する方が良い。


Cargo will automatically ignore files ignored by your version control system when packaging, but if you want to specify an extra set of files to ignore you can use the exclude key in the manifest: Cargoは、パッケージング時にバージョンコントロールシステムで無視されるファイルを自動的に無視しますが、無視するファイルを指定する場合は、マニフェストでexcludeキーを使用できます。



The syntax of each element in this array is what rust-lang/glob accepts.この配列の各要素の構文は、 rust-lang/glob受け入れるものです。 If you'd rather roll with a whitelist instead of a blacklist, Cargo also supports an include key, which if set, overrides the exclude key:ブラックリストの代わりにホワイトリストを使用するのであれば、Cargo includeキーもサポートしています。これが設定されていると、 excludeキーがオーバーライドされます。



Uploading the crateクレートのアップロード


Now that we've got a *.crate file ready to go, it can be uploaded to crates.io with the cargo publish command. *.crateファイルを用意したら、 cargo publishコマンドでcrates.ioにアップロードすることができます。 And that's it, you've now published your first crate!それで、あなたは今、あなたの最初の木枠を公開しました！



If you'd like to skip the cargo package step, the cargo publish subcommand will automatically package up the local crate if a copy isn't found already. cargo packageステップをスキップしたい場合は、コピーがまだ見つからない場合は、 cargo publishサブコマンドが自動的にローカルクレートをパッケージ化します。


Be sure to check out the metadata you can specify to ensure your crate can be discovered more easily!あなたのクレートがより簡単に発見できるように指定できるメタデータを必ずチェックしてください！


Publishing a new version of an existing crate既存のクレートの新しいバージョンを公開する


In order to release a new version, change the version value specified in your Cargo.toml manifest.新しいバージョンをリリースするには、 Cargo.tomlマニフェストで指定されたversion値を変更します。 Keep in mind the semver rules . セーバールールを覚えておいてください。 Then optionally run cargo package if you want to inspect the *.crate file for the new version before publishing, and run cargo publish to upload the new version.次に、 cargo publish前に新しいバージョンの*.crateファイルを調べる場合はcargo package実行し、新しいバージョンをアップロードするにはcargo publishを実行します。


Managing a crates.io-based crate crates.ioベースのクレートの管理


Management of crates is primarily done through the command line cargo tool rather than the crates.io web interface.クレートの管理は、主にcrates.io Webインターフェイスではなく、コマンドラインのcargoツールを使用して行われます。 For this, there are a few subcommands to manage a crate.このために、クレートを管理するためのいくつかのサブコマンドがあります。


cargo yank


Occasions may arise where you publish a version of a crate that actually ends up being broken for one reason or another (syntax error, forgot to include a file, etc.).実際に何らかの理由で壊れてしまうようなクレートのバージョンを公開した場合（文法エラー、ファイルの忘れなど）が発生することがあります。 For situations such as this, Cargo supports a “yank” of a version of a crate.このような場合、Cargoはクレートのバージョンの「ヤンク」をサポートします。



A yank does not delete any code.ヤンクはコードを削除しません 。 This feature is not intended for deleting accidentally uploaded secrets, for example.この機能は、誤ってアップロードされた機密情報を削除するためのものではありません。 If that happens, you must reset those secrets immediately.そのような場合は、すぐにそれらの秘密をリセットする必要があります。


The semantics of a yanked version are that no new dependencies can be created against that version, but all existing dependencies continue to work.ヤンクされたバージョンのセマンティクスは、そのバージョンに対して新しい依存関係を作成することはできませんが、既存のすべての依存関係は引き続き機能します。 One of the major goals of crates.io is to act as a permanent archive of crates that does not change over time, and allowing deletion of a version would go against this goal. crates.ioの主な目標の1つは、時間の経過とともに変化しない箱の永久アーカイブとして機能し、バージョンの削除がこの目標に反して行かれることです。 Essentially a yank means that all projects with a Cargo.lock will not break, while any future Cargo.lock files generated will not list the yanked version.基本的にヤンクとは、 Cargo.lockを持つすべてのプロジェクトが中断しないことをCargo.lockます。生成される将来のCargo.lockファイルは、ヤンクされたバージョンをリストしません。


cargo owner


A crate is often developed by more than one person, or the primary maintainer may change over time!一人のクレートが複数の人によって開発されることが多く、プライマリメンテナーが時間の経過とともに変化する可能性があります。 The owner of a crate is the only person allowed to publish new versions of the crate, but an owner may designate additional owners.クレートの所有者は、クレートの新しいバージョンを公開することができる唯一の人ですが、所有者は追加の所有者を指定することができます。



The owner IDs given to these commands must be GitHub user names or GitHub teams.これらのコマンドに与えられた所有者IDは、GitHubユーザー名またはGitHubチームでなければなりません。


If a user name is given to --add , that user becomes a “named” owner, with full rights to the crate.ユーザー名が--addに指定されている場合、そのユーザーは「名前付き」の所有者になり、クレートに対する完全な権利が与えられます。 In addition to being able to publish or yank versions of the crate, they have the ability to add or remove owners, including the owner that made them an owner.クレートのバージョンを公開またはヤンクできることに加えて、所有者に所有者を加えた所有者を含めて 、所有者を追加または削除することができます 。 Needless to say, you shouldn't make people you don't fully trust into a named owner.言うまでもなく、名前のついた所有者を完全に信用していない人を作るべきではありません。 In order to become a named owner, a user must have logged into crates.io previously.名前付き所有者になるためには、以前にcrates.ioログインしていなければなりません。


If a team name is given to --add , that team becomes a “team” owner, with restricted right to the crate.チーム名が--addに与えられた場合、そのチームは「チーム」オーナーになり、クレートの制限付き権利が与えられます。 While they have permission to publish or yank versions of the crate, they do not have the ability to add or remove owners.彼らは、クレートのバージョンを発行またはヤンクする権限を持っていますが、所有者を追加または削除する機能はありません 。 In addition to being more convenient for managing groups of owners, teams are just a bit more secure against owners becoming malicious.オーナーのグループを管理するのに便利なだけでなく、チームは所有者が悪意を持ってしまうのを防ぐために少しだけ安全です。


The syntax for teams is currently github:org:team (see examples above).チームの構文は現在github:org:team （上記の例を参照）。 In order to add a team as an owner one must be a member of that team.チームをオーナーとして追加するには、そのチームのメンバーでなければなりません。 No such restriction applies to removing a team as an owner.所有者としてのチームの削除には、そのような制限は適用されません。

It looks like you don't have permission to query a necessary property from GitHub to complete this request.このリクエストを完了するためにGitHubから必要なプロパティを照会する権限がありません。 You may need to re-authenticate on crates.io to grant permission to read GitHub org memberships. GitHub組織のメンバーシップを読む権限を与えるには、 crates.io再認証が必要な場合があります。 Just go to https://crates.io/login https://crates.io/loginにアクセスしてください

GitHubのチームアクセス制御のサポートはエンタープライズグレードです。
