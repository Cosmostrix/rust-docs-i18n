Wrapping errors折り返しエラー


An alternative to boxing errors is to wrap them in your own error type.ボクシングエラーの代わりに、あなた自身のエラータイプでそれらをラップすることです。


We will defer to the parse error implementation for their error.私たちは、エラーの解析エラーの実装を延期します。 Supplying extra info requires adding more data to the type.余分な情報を入力するには、そのデータ型にデータを追加する必要があります。
This is a wrapper, so defer to the underlying types' implementation of fmt .これはラッパーなので、基になる型のfmtの実装を延期します。
This already impls Error , so defer to its own implementation.これはすでにError意味しているので、独自の実装を延期します。
The cause is the underlying implementation error type.原因は、基本となる実装エラータイプです。 Is implicitly cast to the trait object &error::Error .暗黙的に特性オブジェクト&error::Errorキャストされます。 This works because the underlying type already implements the Error trait.これは、基になる型が既にError特性を実装しているために機能します。
Implement the conversion from ParseIntError to DoubleError . ParseIntErrorからDoubleErrorへの変換を実装しDoubleError 。 This will be automatically called by ?これは自動的に? if a ParseIntError needs to be converted into a DoubleError . ParseIntError DoubleError変換する必要がある場合。

This adds a bit more boilerplate for handling errors and might not be needed in all applications.これにより、エラーを処理するための定型文が少し追加され、すべてのアプリケーションで必要とされることはありません。 There are some libraries that can take care of the boilerplate for you.ボイラープレートの世話をするライブラリがいくつかあります。
